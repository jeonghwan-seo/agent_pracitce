# Commit Convention

## 커밋 메시지 구조

```
<type> <subject>

<body>

<footer>
```

---

## Type 종류

| Type | 설명 |
|------|------|
| `feat` | 새로운 기능 추가 |
| `fix` | 버그 수정 |
| `docs` | 문서 수정 (README 등) |
| `style` | 코드 포맷 변경, 세미콜론 누락 등 (로직 변경 없음) |
| `refactor` | 코드 리팩토링 (기능 변경 없음) |
| `test` | 테스트 코드 추가 또는 수정 |
| `chore` | 빌드 설정, 패키지 매니저 설정 등 기타 변경 |

---

## 규칙

1. **커밋 메시지는 한글**로 작성합니다.
2. **헤더(type)는 `< >` 문자**로 감싸서 표기합니다. (예: `<feat>`, `<fix>`)
3. **subject**는 50자 이내로 작성합니다.
4. **subject** 끝에 마침표(`.`)를 붙이지 않습니다.
5. **body**는 선택 사항이며, 변경 이유와 내용을 설명합니다.
6. **body**와 **subject** 사이에는 빈 줄을 하나 넣습니다.
7. **footer**는 선택 사항이며, 관련 이슈 번호를 기재합니다. (예: `Closes #123`)

---

## 예시

```
<feat> 로그인 기능 추가

사용자 이메일/비밀번호 기반 로그인 기능을 추가합니다.
JWT 토큰을 발급하여 세션을 관리합니다.

Closes #42
```

```
<fix> 긴급 Fix
```

```
<docs> README 설치 가이드 업데이트
```
